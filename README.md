# Provenance Guard

A Flask backend for creative platforms that classifies whether submitted text was
written by a human or generated by AI. It returns a confidence score, a
plain-language transparency label, logs every decision to an append-only audit
trail, and lets creators contest misclassifications through an appeals workflow.

---

## Quick start

```bash
git clone https://github.com/your-username/provenance-guard
cd provenance-guard
pip install -r requirements.txt
cp .env.example .env        # add your GROQ_API_KEY
python app.py
```

```bash
# Submit text for classification
curl -s -X POST http://localhost:5000/submit \
  -H "Content-Type: application/json" \
  -d '{"text": "The sun dipped below the horizon...", "creator_id": "alice"}' \
  | python -m json.tool

# Appeal a classification
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "<uuid from submit>", "creator_reasoning": "I wrote this myself."}' \
  | python -m json.tool

# View the audit log
curl -s "http://localhost:5000/log?limit=10" | python -m json.tool
```

## Architecture overview

A submitted piece of text takes this path through the system:

```
POST /submit
{ text, creator_id? }
        │
        ▼
  [Rate Limiter]
  Flask-Limiter
  10/min · 100/day per IP
        │
        ▼
   ┌────────────────────────────────────────┐
   │              raw text                  │
   ▼                                        ▼
[Signal 1]                           [Signal 2]
LLM Classifier                       Stylometric Analyzer
detector.py                          detector_stylo.py
Groq llama-3.3-70b-versatile         Pure Python heuristics
   │                                        │
   │ llm_score (0.0–1.0)                    │ stylo_score (0.0–1.0)
   └──────────────────┬─────────────────────┘
                      │
                      ▼
           [Confidence Scorer]  scorer.py
           ai_probability = (0.60 × llm_score) + (0.40 × stylo_score)
           classification  = "human" | "ai" | "uncertain"
           confidence_level = "high" | "uncertain"
                      │
                      ▼
           [Label Generator]  scorer.py
           selects one of three label variants
                      │
                      ▼
           [Audit Log]  auditor.py
           append JSON line → logs/audit.jsonl
                      │
                      ▼
           HTTP 200 JSON response
```

**Appeal flow:**

```
POST /appeal
{ content_id, creator_reasoning }
        │
        ▼
  look up content_id in audit log
  404 if not found · 409 if appeal already exists
        │
        ▼
  append appeal record → logs/audit.jsonl
        │
        ▼
  HTTP 200 { appeal_id, content_id, status: "under_review", message }
```

The two flows are fully independent — an appeal only touches the audit log, not
the detection pipeline. Re-classification is not automated; the appeal flags the
submission for human review.

## Detection signals

### Signal 1 — LLM classifier (`detector.py`)

**What it measures:** Semantic and stylistic coherence, holistically. The model
reads the text as a reader would — does the voice feel personal or optimized?
Does the phrasing feel reaching for something, or efficient and generic?
Idiosyncratic word choices, tonal consistency, and the presence or absence of
genuine perspective are all things an LLM can assess that no surface metric captures.

**How it works:** The text is sent to Groq (`llama-3.3-70b-versatile`) at
temperature 0.1. The prompt asks for a JSON object with a `verdict` (`"human"`
or `"ai"`) and a `raw_confidence` (0.0–1.0 indicating certainty in that verdict).
The score is then normalized so 0.0 always means confident human and 1.0 always
means confident AI, regardless of which verdict was returned.

**Why this signal:** AI models are better than any fixed heuristic at recognizing
patterns like hedging language, uniform sentence rhythm, and the absence of
genuine first-person specificity. These are exactly the properties that distinguish
AI-generated prose from human writing with an unusual style, which is the hard case
the system has to handle fairly.

**What it misses:** Short texts (under 100 words), where there isn't enough voice
to assess. Clean, experienced human writers whose prose happens to be smooth and
efficient. Any text where the submitted content has been heavily edited after
generation, which can erase the LLM's fingerprints at the semantic level.

---

### Signal 2 — Stylometric analyzer (`detector_stylo.py`)

**What it measures:** Four structural surface properties of the text, each
normalized to a 0–1 score where 1.0 = AI-like, then averaged into a single
`stylo_score`.

| Metric | Human range | AI range | What it captures |
|--------|-------------|----------|------------------|
| Sentence length variance (std dev) | std ≥ 30 → score 0.0 | std ≤ 5 → score 1.0 | AI prose has unnaturally uniform sentence lengths |
| Type-token ratio (unique/total words) | TTR ≥ 0.90 → score 0.0 | TTR ≤ 0.60 → score 1.0 | AI models reuse vocabulary more than humans |
| Punctuation density (per 100 words) | density ≥ 8 → score 0.0 | density ≤ 2 → score 1.0 | Human writing uses more expressive punctuation |
| Average word length (chars) | awl ≤ 4.0 → score 0.0 | awl ≥ 6.5 → score 1.0 | AI text skews toward Latinate, multisyllabic vocabulary |

Texts under 50 words have their score pulled toward 0.5 proportionally
(`score = 0.5 + (raw_score - 0.5) × word_count/50`). Variance and TTR are
meaningless with 2–3 sentences; dampening is more honest than a false high-confidence
score on a haiku.

**Why this signal:** It is completely independent of the LLM signal — one is
semantic, one is structural. They can genuinely disagree. When they do, the
disagreement is itself informative: lightly edited AI output tends to fool the
structural signal (it's been touched enough to add punctuation and vary sentence
length) while the LLM still catches the residual voice patterns. That disagreement
correctly produces an uncertain verdict rather than a wrong confident one.

**Why pure Python:** No API calls, no rate limits, no latency. Every submission
runs the stylometric analysis regardless of whether Groq is reachable, and the
result is deterministic and inspectable.

**What it misses:** Legal, academic, and technical writing, where formal register
and Latinate vocabulary are conventions of the genre, not evidence of AI generation.
Deliberately repetitive literary prose (anaphora, refrains), where the low TTR looks
like AI vocabulary reuse. Anything under 50 words, where the dampening makes the
signal nearly neutral.

## Confidence scoring

### Formula

```
ai_probability = (0.60 × llm_score) + (0.40 × stylo_score)
```

The LLM gets 60% weight because it reads holistic semantic patterns — tone, voice,
whether the text feels like it's reaching for something — that the stylometric signal
can't access. On the cases that matter most (formal human writing that looks AI
structurally, lightly edited AI that reads smoothly), the LLM is more informative.
Stylometrics get 40% as a structural cross-check that is immune to prompt
manipulation of the LLM.

### Classification thresholds

| `ai_probability` | `classification` | `confidence_level` |
|------------------|------------------|--------------------|
| ≥ 0.80 | `ai` | `high` |
| ≤ 0.20 | `human` | `high` |
| 0.21 – 0.79 | `uncertain` | `uncertain` |

The thresholds are intentionally asymmetric. A false positive — calling a human
creator's work AI-generated — is reputational harm to a real person. The system
only commits to `ai / high` when both signals are leaning AI fairly hard. It would
rather say "uncertain" than accuse. This asymmetry is a deliberate design choice,
not a limitation.

### Validating that scores are meaningful

The test suite in `examples.py` runs 10 examples spanning the full range and
verifies that scores move in the right direction. A few anchoring observations from
running the corpus:

- Stream-of-consciousness human writing: `llm=0.05, stylo=0.36 → ai_probability=0.17`
- Casual chat with slang: `llm=0.08, stylo=0.31 → ai_probability=0.17`
- Lightly edited AI output: `llm=0.65, stylo=0.42 → ai_probability=0.56`
- Classic AI corporate prose: `llm=0.92, stylo=0.53 → ai_probability=0.77`

The scores are not symmetric around 0.5 by accident. The human examples cluster
below 0.20 (reaching the high-confidence human threshold), while the AI examples
cluster in the 0.55–0.80 range — rarely crossing 0.80, because the stylometric
signal on real AI text is moderate enough to pull the fusion score below the
threshold. That compression toward "uncertain" is the system working correctly,
not a calibration failure.

### Two contrasting submissions

**High-confidence human** — casual restaurant review:

```
Text: "ok so i finally tried that new ramen place downtown and honestly?
underwhelming. the broth was fine but they put WAY too much sodium in it and
i was thirsty for like three hours after. my friend got the spicy version and
said it was better. probably wont go back unless someone drags me there lol.
also the line was insane, like 45 minutes on a tuesday? who has time for that"

  llm_score      : 0.08   (LLM confident human — recognizes register, slang, named friend)
  stylo_score    : 0.3076 (awl=4.1, dense punctuation, irregular sentences)
    sentence_length_variance : 6.04
    type_token_ratio         : 0.833
    punctuation_density      : 9.72
    avg_word_length          : 4.12
  ai_probability : 0.171
  classification : human / high
  label          : ✅ Human-Authored
```

**Lower-confidence, borderline** — polished personal essay:

```
Text: "Climate change is the defining challenge of our era, yet our political
response remains hopelessly fragmented. I grew up near a river that flooded
twice in my childhood; it flooded six times in the last decade. That is not
an abstraction. What frustrates me most is not denialism — that is a fringe
position now — but the comfortable delay of people who accept the science and
still find reasons to wait. The transition costs are real. So is the cost of
doing nothing, and unlike transition costs, that one compounds."

  llm_score      : 0.18   (LLM leans human — first-person anecdote, rhetorical specificity)
  stylo_score    : 0.4050 (moderate — semicolons and dashes help, but prose is polished)
    sentence_length_variance : 8.39
    type_token_ratio         : 0.753
    punctuation_density      : 11.24
    avg_word_length          : 4.66
  ai_probability : 0.270
  classification : uncertain / uncertain
  label          : 🔍 Authorship Uncertain
```

The difference between 0.171 and 0.270 is the difference between a clear human
signal (slang, all-lowercase, rhetorical questions) and a merely plausible human
signal (personal anecdote, but still polished prose). Both are probably human.
The second one lands in "uncertain" rather than "human / high" because the
system only commits when it's confident — a creator who writes well shouldn't
be penalized, but the system also shouldn't promise certainty it doesn't have.

## Transparency labels

All three variants are returned as the `label_text` field in `/submit` responses.
These are the exact strings the system produces.

**High-confidence AI** (`ai_probability ≥ 0.80`):

```
⚠️ AI-Generated Content
Our analysis strongly suggests this work was generated by an AI writing tool, not
written by the credited creator. Confidence: High. If you are the creator and believe
this is incorrect, you may contest this classification using the appeals process.
```

**High-confidence Human** (`ai_probability ≤ 0.20`):

```
✅ Human-Authored
Our analysis strongly suggests this work was written by the credited creator.
Confidence: High.
```

**Uncertain** (`0.21 ≤ ai_probability ≤ 0.79`):

```
🔍 Authorship Uncertain
Our system could not confidently determine whether this work was written by a human
or generated by AI. This may reflect an unusual writing style, a short submission,
or signals that disagree. The creator may contest this classification. This label
may be updated following a review.
```

The uncertain label explicitly names three causes ("unusual writing style, a short
submission, or signals that disagree") because creators deserve to understand why
they were flagged, not just that they were. It also states the label can be
updated — because it can, via the appeal flow.

## Rate limiting

`POST /submit` is limited to **10 requests per minute** and **100 requests per day**
per IP address, enforced by Flask-Limiter with in-memory storage.

```python
@limiter.limit("10 per minute;100 per day")
def submit():
    ...
```

**Why 10 per minute:** A creator using the platform interactively — uploading a
poem, reading the result, revising — might submit 3–4 times in a session. 10/min
is generous for that use case while being far too slow for a script. A simple loop
sending 10 requests takes 1 second; the 11th gets a 429. That stops automated
flooding at almost no cost to legitimate users.

**Why 100 per day:** The Groq free tier has daily token limits. Each submission
triggers a real API call. 100 submissions per IP per day is realistic for any
individual creator — it would take deliberate effort to hit that legitimately —
while preventing a single actor from exhausting the quota for everyone else or
filling the audit log with synthetic entries to pollute the data.

**Verification:** To confirm rate limiting is working, send 12 rapid requests while
the server is running:

```bash
for i in $(seq 1 12); do
  curl -s -o /dev/null -w "%{http_code}\n" -X POST http://localhost:5000/submit \
    -H "Content-Type: application/json" \
    -d '{"text": "Rate limit test.", "creator_id": "rl-test"}'
done
```

Expected output: 10 lines of `200`, then 2 lines of `429`.

## Audit log

Every submission and appeal is appended to `logs/audit.jsonl` as a newline-
delimited JSON record. The file is append-only, human-readable, and every line
is independently parseable. `GET /log` returns the most recent entries as JSON.

**Submission record:**

## Known limitations

**Legal, academic, and technical writing will frequently land in "uncertain" when
it should land in "human / high."**

The stylometric signal flags Latinate vocabulary, uniform sentence structure, and
lower punctuation density as AI-like — and formal registers share all three of
those properties for genre reasons, not because the text was generated. A lawyer's
brief, a scientific abstract, or a philosophy paper will score 0.55–0.65 on the
stylometric signal even when entirely human-written. The LLM typically scores
these around 0.50–0.60 as well, because it has been trained on enormous amounts
of formal human writing that looks exactly like AI writing structurally.

The result: the system will systematically under-serve creators who write formally.
Their work will be flagged as uncertain rather than confidently human, forcing them
to appeal classifications that were never confident to begin with.

The fix would require either a genre classifier upstream (route formal-register
text to a different signal set) or training the stylometric thresholds separately
per genre. Neither is in scope here — but the appeals flow exists precisely for
this scenario, and the "uncertain" label explicitly acknowledges that "unusual
writing style" is a recognized cause of the uncertainty.

## Spec reflection

**Where the spec helped:** The instruction to design the three label variants
before writing any code — in Milestone 2, as plain text — was the single most
useful constraint in the project. Doing that forced a decision about what "uncertain"
actually means to a non-technical user before there was any implementation pressure
to simplify it. The final label text ("This may reflect an unusual writing style,
a short submission, or signals that disagree") came directly from that early draft,
unchanged. Without the spec's insistence on writing it first, that specificity would
almost certainly have been cut in favor of something vaguer.

**Where implementation diverged:** The spec describes the audit log as a tool for
grading visibility — a way to show decision history — and treats `GET /log` as
roughly equivalent to tailing the file. In implementation, the audit log ended up
doing more structural work than anticipated: the appeals flow queries it directly
to verify a submission exists (`find_submission()`) and check for duplicate appeals
(`find_appeal()`), with 404 and 409 error handling that depends on those lookups.
That makes `logs/audit.jsonl` the de facto state store for the entire system, not
just a log. That wasn't in the spec — it emerged from realizing that adding a
separate in-memory dict for submission state would mean appeals filed after a server
restart couldn't find their original submission. The JSONL log is durable; an
in-memory dict isn't. The divergence was the right call.

## AI usage

### Instance 1 — Flask app skeleton and rate limiter wiring

The M5 `app.py` was initially generated by providing Claude the architecture diagram
(the ASCII flow), the spec sections on the appeal workflow and rate limiting, and
the existing M3/M4 route signatures. The generated output wired Flask-Limiter
correctly and produced the right `429` handler, but the initial appeal endpoint
did not include the 409 guard for duplicate appeals — it accepted a second appeal
for the same `content_id` and appended a second record silently. That was revised
manually: `find_appeal()` was added to `auditor.py` and the conflict check was
added to the route before the `auditor.append_appeal()` call. The reason the AI
missed it is that the spec only says "capture the creator's reasoning and update
status" — it doesn't explicitly say what to do when the same creator appeals twice.
That edge case required a judgment call the spec left open.
